L0 · World Events
Зовнішній світ — єдине джерело подій для NPC
реалізовано
Що це таке
World Engine генерує події і публікує їх у Redis черги. Celery worker NPC забирає подію і передає вгору по шарах. Це єдина точка входу — все що відбувається у світі приходить спочатку сюди.
Структура об'єкта події
event.typethreat / social / economic / environmental / internal
event.severity0.0 – 1.0 · об'єктивна сила події у світі
event.tagsthreat:physical · injustice · public · witness:npc_id…
event.source_npcID хто ініціював (або null якщо стихія)
event.priorityHIGH / MEDIUM / LOW → визначає чергу Redis
Що отримує / віддає
Отримує від
World EnginePlayer actionsTick scheduler
Віддає в
event object → Redis queue → L1/L2
Файли цього шару
world_engine/engine.pyГоловний клас World Engine. Керує часом, погодою, економікою, генерує подіїє
world_engine/event_factory.pyСтворює event об'єкти з правильною структурою і тегамиє
infrastructure/redis_client.pyКлієнт Redis. pub/sub + черги HIGH/MEDIUM/LOWє
worker/celery_app.pyКонфігурація Celery. Кожен NPC = окремий worker processє
Event object — те що летить в Redis
# Redis queue: MEDIUM_PRIORITY
# key: npc:aria_001:events

{
  "event_id":     "evt_8472a",
  "type":         "threat",
  "subtype":      "physical",
  "severity":     0.85,
  "tags":         ["threat:physical", "injustice", "public"],
  "source_npc":   "boris_002",
  "target_npc":   "aria_001",
  "location":     "market_square",
  "witness_npcs": ["oleh_003"],
  "tick":         2847,
  "priority":     "MEDIUM"
}
severity — об'єктивна сила події. Не плутати з persona.sensitivity — це різні речі. Один і той самий удар (severity=0.85) різні NPCі сприймають по-різному через sensitivity персон.

L1 · Body
Фізичний стан — змінюється автономно кожен тік
описано
Що це таке
Тіло існує незалежно від подій. Hunger і fatigue накопичуються самі. Stress залишається після загрози і повільно спадає. Без цього шару NPC завжди у "нейтральному" стані — як манекен.
Дані
body.hunger0.0–1.0 · росте кожен тік. При >0.7 → Instinct +0.4
body.fatigue0.0–1.0 · знижує attention_budget в L2
body.stress0.0–1.0 · прямий вхід в Instinct raw score
body.health0.0–1.0 · впливає на базову чутливість всіх персон
Що отримує / віддає
Отримує від
tick clock (автономно)фізичні події (удар, їжа)
Віддає в
stress → L3 Instinctfatigue → L2 attention_budgethunger → L3 Instinct
⚠ Не реалізовано
body state зараз статичний. Aria завжди однаково уважна і не голодна. Без цього шару всі NPCі реагують ідентично в 6 ранку і в 10 вечора.
Файли
npc/layers/body_state.pyКлас BodyState. tick() метод що оновлює hunger/fatigue/stress кожен тіквідсутній
npc/models/body_model.pySQLAlchemy модель для зберігання body state в PostgreSQLвідсутній
Body state об'єкт — передається в L2 і L3
# PostgreSQL: table npc_body_state
# оновлюється кожен тік

{
  "npc_id":   "aria_001",
  "tick":     2847,
  "hunger":   0.42,   # не їла 4 години ігрового часу
  "fatigue":  0.31,   # помірна втома
  "stress":   0.68,   # залишок від сварки 3 тіки тому
  "health":   0.95
}

# → stress=0.68 передається в L3 Instinct:
# instinct.raw += 0.68 * 0.85  =  +0.578

L2 · Perception Filter
Що NPC взагалі помічає — увага як обмежений ресурс
відсутній
Що це таке
Найважливіший відсутній шар. Зараз NPC отримує і реагує на всі події однаково. Але attention_budget — обмежений ресурс. Стомлена Aria не помічає тонких сигналів. Голодна — фільтрує все крім їжі.
Дані (яких немає)
attention.budget= 1.0 − fatigue. Визначає скільки подій може обробитивідсутній
filter.thresholdevent.severity < threshold → подія ігноруєтьсявідсутній
filter.biasголодний → знижує threshold для food-тегіввідсутній
⚠ Чому критично
Без L2 — NPC реагує як сервер що обробляє всі запити з однаковим пріоритетом. З L2 — Aria в стресі може не помітити що клієнт чекає, бо вся увага на Борисі. Це і є різниця між "обчисленням" і "живою людиною".
Файли (яких немає)
npc/layers/perception.pyКлас PerceptionFilter. filter_events(events, body_state) → filtered_eventsвідсутній
npc/layers/attention.pyРахує attention_budget з fatigue. Визначає threshold і biasвідсутній
Як мав би виглядати фільтр
# Вхід: всі події тіку + body state
# Вихід: тільки ті що пройшли фільтр

input:
  body.fatigue   = 0.70   # втомлена
  body.hunger    = 0.80   # дуже голодна
  attention_budget = 0.30  # = 1.0 - fatigue
  threshold      = 0.60   # пропускає тільки severity > 0.60

events:
  evt_A: severity=0.85, tags=["threat:physical"]  → ✅ пройшов
  evt_B: severity=0.30, tags=["social"]           → ❌ проігноровано
  evt_C: severity=0.25, tags=["food_nearby"]      → ✅ bias знизив threshold

output → L3:  [evt_A, evt_C]

L3 · MCS Core
4 персони · softmax · delta — серце системи
описано
Що це таке
Ядро MCS. Отримує відфільтровані події і body.stress, запускає 4 персони незалежно. Кожна рахує raw_score, потім softmax перетворює їх у ваги. Delta визначає наскільки змінився стан.
Дані
personality.base_traits{protector:0.65, instinct:0.55, thinker:0.70, mask:0.45}
state.current_weights{protector:0.42, instinct:0.28, thinker:0.18, mask:0.12}
state.delta_levelLOW / MID / HIGH / EXPLOSIVE
state.emotional_stateCALM / ALERT / TENSE / EXPLOSIVE ← з домінуючої персони
decay.rateяк швидко ваги повертаються до base_traits без подійвідсутній
Що отримує / віддає
Отримує від
filtered events ← L2body.stress ← L1memory.incidents
Віддає в
weights → L4delta_level → L4emotional_state → L5
Файли
npc/core/tick_processor.pyГоловний orchestrator тіку. Викликає всі персони, softmax, delta. Метод process_tick(event, body)відсутній
npc/core/persona_engine.py4 класи персон. Кожна має метод score(event, body) → floatвідсутній
npc/core/softmax.pysoftmax(scores: dict) → weights: dict. Чиста математика.відсутній
npc/core/delta.pycalculate_delta(current, previous) → float + DeltaLevel enumвідсутній
npc/models/npc_state.pySQLAlchemy модель. Зберігає weights, delta, emotional_state в PostgreSQLвідсутній
Вхід і вихід tick_processor
# ВХІД в tick_processor.process_tick()
event:   { type: "threat", severity: 0.85, tags: ["threat:physical", "injustice"] }
body:    { stress: 0.68, fatigue: 0.31, hunger: 0.42 }

# КРОК 1: raw scores
raw:
  protector = 0.65 + 0.85×1.0 + 0.85×0.53 = 2.10
  instinct  = 0.55 + 0.68×0.85          = 1.13
  thinker   = 0.70                         = 0.70
  mask      = 0.45 + 1×0.15              = 0.60

# КРОК 2: softmax → weights
weights:
  protector = 0.42, instinct = 0.28
  thinker   = 0.18, mask     = 0.12   Σ = 1.00 ✓

# КРОК 3: delta vs попередній тік
previous:  { protector: 0.27, instinct: 0.24, thinker: 0.29, mask: 0.20 }
delta:     |0.42-0.27|+|0.28-0.24|+|0.18-0.29|+|0.12-0.20| = 0.38
level:     HIGH  (0.25 < 0.38 < 0.50)

# ВИХІД → L4 Decision Gate
{
  "weights":         { protector: 0.42, instinct: 0.28, ... },
  "delta_level":     "HIGH",
  "emotional_state": "TENSE"
}

L4 · Decision Gate
LLM чи ні — ключова розвилка. Дорого vs дешево.
описано
Що це таке
Decision Gate дивиться на delta_level і вирішує що далі. LLM викликається тільки при EXPLOSIVE або коли NPC говорить з гравцем. Всі інші випадки — детерміновані.
Пороги рішень
EXPLOSIVE > 0.50LLM виклик негайно → генерувати повну відповідь
HIGH 0.25–0.50Записати стан, MEDIUM priority тік, NPC пам'ятає але мовчить
MID 0.10–0.25Записати стан, LOW priority тік
LOW < 0.10Тихо оновити weights, наступний тік по розкладу
llm.context_builderщо саме передається в LLM: стан + пам'ять + стосунки?відсутній
llm.response_parserяк LLM відповідь → конкретна дія (speak/move/attack)відсутній
Файли
npc/layers/decision_gate.pydecide(delta_level, weights) → Action або None. Головна логіка розгалуженнявідсутній
npc/llm/context_builder.pyБудує LLM prompt з NPC стану, пам'яті, стосунків. Найважливіший невизначений файл.відсутній
npc/llm/response_parser.pyПарсить LLM відповідь → структурований Action об'єктвідсутній
npc/memory/writer.pyЗберігає стан після кожного тіку в PostgreSQLвідсутній

L5 · Action
Що NPC робить у світі — вихід всієї системи
відсутній
Що це таке
Вихідний шар. Сюди приходить або LLM результат або детермінована реакція. Action визначає що Aria робить: говорить, мовчить, рухається, торгує. Без цього шару NPC може "думати" скільки завгодно — але нічого не відбувається у світі.
Типи дій (не визначено)
action.speakговорить щось · target + content + toneвідсутній
action.moveпереміщується · destinationвідсутній
action.tradeініціює / відхиляє торгівлювідсутній
action.ignoreсвідомо ігнорує · теж є діявідсутній
social_signalщо емітується в Social Fabric → інші NPCі чуютьвідсутній
⚠ Блокує взаємодію між NPCями
Без L5 не можна побудувати Social Fabric. NPCі не можуть впливати один на одного бо немає виходу з системи в зовнішній світ.
Файли (яких немає)
npc/layers/action_emitter.pyemit(action) → World Engine + Social Fabric. Головний вихід системи.відсутній
npc/actions/action_types.pyEnum і dataclass для всіх типів дій. speak / move / trade / ignore…відсутній
social_fabric/signal_bus.pyПриймає сигнали від Action і розповсюджує NPCям в радіусівідсутній

Self Model
Як NPC бачить себе і що думає що думають інші
відсутній
Що це таке
Окремий шар даних поверх 4 персон. Не замінює їх — але впливає на base_traits через multiplier. Без Self Model всі NPCі з однаковими base_traits поводяться однаково. З ним — кожен через призму власної ідентичності.
Три осі
self.identity"я сильна" · "я чесна" — список переконань про себевідсутній
self.worthсамооцінка 0–1. LOW → Protector чутливість ×1.5 на образивідсутній
self.world_viewчи світ справедливий? безпечний? — впливає на Protector/Thinker baseвідсутній
perceived.by_borisщо Aria ДУМАЄ що Борис думає про неї. Може бути неправдою.відсутній
perceived.desiredякою Aria ХОЧЕ виглядати. Конфлікт з perceived → Mask ↑відсутній
⚠ Найбільша архітектурна дірка
Aria з self.worth=0.3 ("я слабка, мене не поважають") і Aria з self.worth=0.9 ("я впевнена господиня ринку") — це два абсолютно різних NPC навіть з однаковими base_traits.
Файли (яких немає)
npc/self_model/model.pyКлас SelfModel з identity, worth, world_view, perceived dictвідсутній
npc/self_model/modifier.pyapply_self_model(base_traits, self_model) → modified_traits. Мультиплікатори.відсутній
npc/self_model/updater.pyОновлює self_model під час memory consolidation ("сон"). Повільні зміни.відсутній
