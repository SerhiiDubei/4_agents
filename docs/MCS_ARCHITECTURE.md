## Суть проекту

NPC не є реактивними персонажами що чекають на гравця.
Вони **живуть автономно** — мають внутрішній стан, переживають події, взаємодіють між собою.
Гравець заходить у світ що вже існує і продовжує існувати після його виходу.

**Ключова цінність:**
- Супер реіграбельність — кожна сесія застає NPCів в іншому стані
- Відчуття реальної людини по той бік — бо поведінка непередбачувана але не рандомна
- Світ що живе сам по собі

## Три системи

```
┌─────────────────────────────────────────────┐
│              WORLD ENGINE                    │  ← що відбувається в світі
│   події, час, погода, економіка, смерті      │
└────────────────────┬────────────────────────┘
                     │ події
┌────────────────────▼────────────────────────┐
│                   MCS                        │  ← як NPC це переживає
│   4 персони, softmax, delta, пам'ять         │  ← БУДУЄМО ПЕРШИМ
└────────────────────┬────────────────────────┘
                     │ дії / сигнали
┌────────────────────▼────────────────────────┐
│             SOCIAL FABRIC                    │  ← як NPCі чують одне одного
│   стосунки, комунікація, чутки, довіра       │
└─────────────────────────────────────────────┘
```

**MCS — фундамент.** World Engine і Social Fabric будуються поверх нього.

## MCS — чому 4 персони

1 персона = примітивна реакція
2 персони = технічно мультиагентна але порожня
3 персони = очевидний крок
**4 персони = перший рівень де виникає справжня напруга і непередбачуваність**

| Персона | Питання | Активується |
|---------|---------|-------------|
| **Protector** | Що треба захистити? | загрози, несправедливість |
| **Instinct** | Що я відчуваю? | фізичні стимули, емоції |
| **Thinker** | Які наслідки? | складні ситуації, вибір |
| **Mask** | Що від мене очікують? | соціальний контекст |

Ніхто не перемагає повністю — домінує, але інші шепочуть.
Це і є джерело непередбачуваності що відчувається як жива людина.

## Серверна архітектура

```
┌──────────────────────────────────────────────┐
│                   Redis                       │
│  pub/sub — спільна реальність                 │
│  черги — high / medium / low priority         │
└───────────┬──────────────────────────────────┘
            │
┌───────────▼──────────────────────────────────┐
│                  Celery                       │
│  Worker NPC_A | Worker NPC_B | Worker NPC_C   │
│  кожен NPC — окремий процес                   │
└───────────┬──────────────────────────────────┘
            │
┌───────────▼──────────────────────────────────┐
│             PostgreSQL / SQLite               │
│  persistent memory, стосунки, history         │
└───────────┬──────────────────────────────────┘
            │
┌───────────▼──────────────────────────────────┐
│                  FastAPI                      │
│  інтерфейс для World Engine + гравця          │
└──────────────────────────────────────────────┘
```

**LLM — локальний на тому ж сервері.** Latency мінімальна, cost не проблема.

## Два рівні тіку (критично важливо)

Найбільший bottleneck — LLM на кожен тік при багатьох NPCях.

```
Рівень 1 — ДЕШЕВИЙ (часто, завжди)
  → чиста математика: softmax scoring, delta, ваги персон
  → без LLM, мілісекунди
  → відбувається постійно

Рівень 2 — ДОРОГИЙ (рідко, по тригеру)
  → LLM викликається тільки коли:
     - NPC реально говорить з кимось
     - delta = EXPLOSIVE (шоковий стан)
     - раз на N тіків для memory consolidation ("сон")
     - реакція на гравця (high priority)
```

---

## Пріоритети черг

```
HIGH    — реакція на гравця (миттєво)
MEDIUM  — world events (швидко)
LOW     — idle life тіки (фоново, коли є ресурс)
```

NPCі живуть навіть при низькому навантаженні — просто повільніше.

---

## Проактивність vs Реактивність

**Важливо:** поточний MCS спек — реактивна система (подія прийшла → персони відреагували).
Але NPC що живе своїм життям має бути **проактивним** — сам ініціює дії.

Потрібно додати в MCS механіку:
- NPC сам вирішує з ким поговорити
- NPC сам ініціює переміщення, дії
- Внутрішній "імпульс" що генерується без зовнішньої події

**Статус:** ще не реалізовано, потребує рішення в спеці.

## Відкриті питання (потрібні рішення)

- [ ] Як часто NPC тікає коли нічого не відбувається? (event-driven / interval / hybrid)
- [ ] Memory persistence — NPC пам'ятає між сесіями гравця?
- [ ] Mask персона — детальніше пропрацювати тригери
- [ ] Conflict resolution між персонами — як саме резолвиться конфлікт Instinct vs Thinker?
- [ ] Проактивність — механіка де NPC сам ініціює дію

## Порядок розробки

```
1. MCS core engine (математика, без LLM)
   → tick processor, softmax, delta, 4 персони

2. Серверна інфра
   → Redis + Celery + FastAPI
   → один NPC як окремий процес

3. LLM інтеграція
   → whispers, діалог, memory consolidation

4. Social Fabric
   → NPC-to-NPC комунікація

5. World Engine
   → генерація подій в світі

6. Візуалізація
   → HUD, debug, спостереження за живими NPCями
```

## Стек

- **Python** — основна мова
- **FastAPI** — API сервер
- **Redis** — event bus, черги
- **Celery** — task queue, NPC workers
- **PostgreSQL** — persistent state
- **LLM локальний** — на тому ж сервері (cost не проблема)
