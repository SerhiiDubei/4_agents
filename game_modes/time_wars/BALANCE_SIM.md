# Balance simulator (без агентів)

Емуляція балансу та темпу гри на матмоделі STATE: тіки, події (шторм, криза), спрощені дії (кооп/крадіжка) з фіксованими ймовірностями. Без LLM.

## База

- **1 тік = 1 ігрова хвилина** = 60 сек (кожен гравець втрачає 60 сек за тік).
- **B** = стартовий час на гравця (в ігрових хвилинах → у скрипті в секундах: `B_sec = B * 60`).
- **T** = довжина гри в тіках (= ігрових хвилинах).

Умова «гра не миттєво обнуляється»: при B = T без додаткового приросту часу (кооп, коди) усі через T тіків у 0. Тому або B > T, або події/дії дають приріст.

## Що робить симулятор

1. **Тік**: кожен активний гравець втрачає `tick_cost_sec` (за замовчуванням 60).
2. **Події** за розкладом: шторм (−X сек усім), криза (у кого час < порогу — −Y сек).
3. **Дії**: кожні N тіків кожен гравець з ймовірністю `p_coop` робить кооп з випадковим іншим (+30 обом), з `p_steal` — крадіжка (d20 + ті самі payoffs з `constants.py`).

Метрики: `survival_rate`, `final_time_mean/median`, `first_elimination_tick_mean/median`, `survival_rate_2plus`.

## Запуск

З кореня репо:

```bash
# Один набір параметрів, 1000 прогонів (за замовчуванням B=20min, T=20, 1 шторм + 1 криза)
python -m game_modes.time_wars.balance_sim --runs 1000

# Сітка параметрів (B=20/25/30, T=15/20/25, 0/1/2 події, 2 профілі поведінки)
python -m game_modes.time_wars.balance_sim --grid --runs 1000 --out game_modes/time_wars/balance_results.json

# Свої параметри
python -m game_modes.time_wars.balance_sim --runs 500 --B 25 --T 20 --storm-sec -180 --crisis-penalty-sec -120 --p-coop 0.4 --p-steal 0.15
```

## Скільки часу займе

- Один прогон: ~0.5–2 ms (залежить від T та n_players).
- **1000 прогонів** одного набору: **~1–5 сек**.
- **Сітка** (29 наборів × 1000 прогонів): **~1–2 хв**.
- 200 прогонів на набір для швидкої перевірки: **~10–15 сек** для всієї сітки.

Тобто повний сет ап тестів: **менше 5 хв** (написати параметри, запустити сітку, переглянути JSON/вивід).

## Як використовувати результати

- **survival_rate** близько 80–98% при хоча б 2 вцілілих — розумний тиск без катівні.
- **first_elimination_tick_median** показує, коли зазвичай перший вибуває; якщо дуже рано (наприклад 5–8) — зменшити силу подій або збільшити B.
- Якщо при B=20, T=20 і двох подіях survival_rate падає нижче 70% — зменшити `storm-sec` / `crisis-penalty-sec` або збільшити B (наприклад 25 хв).

## Відповіді на питання

| Питання | Відповідь |
|--------|-----------|
| Чи достатньо 20 хв? | При B=20, T=20 без подій усі в 0 через 20 тіків. З 1 шторм −4 хв + 1 криза у сітці survival_rate ~97–99%; з B=20, T=25 і тими ж подіями вже ~10–31%. Тобто **20 хв достатньо для T=20**, для T=25 краще B=25–30. |
| Тік кожні 1 хв? | Так: 1 тік = 1 ігрова хв (60 сек) — зручно і узгоджено з одиницями. |
| Скільки подій? | Для старту достатньо **1 шторм + 1 криза** за гру. Value підбирай по survival_rate (наприклад шторм −3…−5 хв, криза −2…−4 хв залежно від порогу). |
| Value подій? | У межах 15–25% від B×n сумарного «негативу» — відчутний тиск без повного обнулення. Конкретику перевіряй емуляцією. |
